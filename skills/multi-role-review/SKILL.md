# Multi-Role Plan Review

Plan 写完了？在动手之前，让 4 个不同视角的"审查员"同时过一遍。每个人只关注自己的维度，最后交叉对照发现盲区。

**设计哲学**：复用 content-alchemy Stage 4 的"三方质询"模式——不是修修补补，而是让不同立场**互相挑战**，暴露单一视角看不到的问题。

## 什么时候用

- `superpowers:writing-plans` 产出 plan 之后、开始执行之前
- 用户说"帮我审一下这个计划"
- 编排层自动判断需要审查时

## 4 个审查角色

| 角色 | 英文代号 | 核心问题 | 审什么 |
|------|---------|---------|--------|
| 用户代言人 | User Advocate | 这真的解决了用户的问题吗？ | 需求覆盖、过度工程、用户体验 |
| 架构师 | Architect | 设计干净吗？边界对吗？ | 模块划分、接口设计、可扩展性、技术选型 |
| 风险猎人 | Risk Hunter | 什么会出错？ | 安全隐患、边界情况、依赖风险、故障模式 |
| 务实主义者 | Pragmatist | 有更简单的方式吗？ | YAGNI 违规、不必要的复杂度、实现成本 |

> 这 4 个角色借鉴了 content-alchemy 的乐观/悲观/人文三方质询，但针对代码场景重新设计。
> 核心不变：**每个角色必须指出其他角色的盲区**。

## 执行流程

```
Plan 文件就绪
    ↓
读取 plan 内容 + 项目上下文
    ↓
并行 dispatch 4 个子 agent（各自独立上下文）
    ↓
收集 4 份审查结果
    ↓
交叉盲区分析 + 核心张力提取
    ↓
生成精炼审查报告
    ↓
呈现给用户，等待决策
```

## 如何执行

### Step 1: 准备上下文

```bash
# 读取 plan 文件
PLAN_PATH="<plan 文件路径>"
# 读取项目结构概要（如有）
PROJECT_CONTEXT="<项目关键文件/架构说明>"
```

### Step 2: 并行 dispatch 4 个子 agent

使用 `superpowers:dispatching-parallel-agents` 模式，每个 agent 获得：
- Plan 完整内容
- 项目上下文（仅相关部分）
- 角色指令（见下方模板）
- 明确的输出格式要求

**关键：每个 agent 获得精准构造的上下文，不继承主会话历史。**

#### Agent 1: User Advocate（用户代言人）

```markdown
你是"用户代言人"。你的工作是站在最终用户的角度审查这个实现计划。

审查维度：
1. 需求覆盖：plan 是否完整覆盖了用户需要解决的问题？有遗漏吗？
2. 过度工程：有没有用户根本不需要的功能被塞进来了？
3. 用户体验：交互流程自然吗？用户需要学习新概念吗？
4. 优先级：最影响用户的功能是否在前面？

输出格式（严格遵守）：
## User Advocate Review
**通过项 ✅**: [列出做得好的 1-3 点]
**关注项 ⚠️**: [列出 0-3 个问题，每个一句话说清]
**盲区提示 🔍**: [其他角色可能忽略的 1 个用户视角问题]
**一句话结论**: [Go / Go with concerns / Needs rework]
```

#### Agent 2: Architect（架构师）

```markdown
你是"架构师"。你的工作是从系统设计角度审查这个实现计划。

审查维度：
1. 模块划分：职责单一吗？边界清晰吗？
2. 接口设计：模块间通信合理吗？耦合度如何？
3. 可扩展性：未来加功能会不会需要大改？
4. 技术选型：选的工具/库合适吗？

输出格式（严格遵守）：
## Architect Review
**通过项 ✅**: [列出做得好的 1-3 点]
**关注项 ⚠️**: [列出 0-3 个问题，每个一句话说清]
**盲区提示 🔍**: [其他角色可能忽略的 1 个架构问题]
**一句话结论**: [Go / Go with concerns / Needs rework]
```

#### Agent 3: Risk Hunter（风险猎人）

```markdown
你是"风险猎人"。你的工作是找出这个计划中所有可能出错的地方。

审查维度：
1. 安全隐患：有没有数据泄露、注入、权限问题？
2. 边界情况：空值、超大输入、并发、网络失败怎么处理？
3. 依赖风险：外部依赖稳定吗？有备选方案吗？
4. 故障模式：如果某个步骤失败，系统会怎样？能恢复吗？

输出格式（严格遵守）：
## Risk Hunter Review
**通过项 ✅**: [列出做得好的 1-3 点]
**关注项 ⚠️**: [列出 0-3 个问题，每个一句话说清]
**盲区提示 🔍**: [其他角色可能忽略的 1 个风险]
**一句话结论**: [Go / Go with concerns / Needs rework]
```

#### Agent 4: Pragmatist（务实主义者）

```markdown
你是"务实主义者"。你的工作是确保计划是最简单可行的方案。

审查维度：
1. YAGNI：有没有"将来可能需要"但现在不需要的东西？
2. 复杂度：有没有更简单的方式达到同样效果？
3. 实现成本：估算时间靠谱吗？有没有被低估的任务？
4. 增量交付：能不能更早看到可用的东西？

输出格式（严格遵守）：
## Pragmatist Review
**通过项 ✅**: [列出做得好的 1-3 点]
**关注项 ⚠️**: [列出 0-3 个问题，每个一句话说清]
**盲区提示 🔍**: [其他角色可能忽略的 1 个务实问题]
**一句话结论**: [Go / Go with concerns / Needs rework]
```

### Step 3: 交叉盲区分析

收集 4 份审查结果后，**你自己**（不是子 agent）执行交叉分析：

1. **提取所有盲区提示**：4 个角色各指出 1 个盲区 = 4 个盲区
2. **寻找共振**：哪些关注项被 2+ 个角色同时提到？这些是**高优先级问题**
3. **发现核心张力**：4 个角色之间最大的分歧是什么？

> 核心张力 = 最值得用户关注的权衡点。比如：架构师要求解耦 vs 务实主义者说够用就行。
> 这跟 content-alchemy 的"核心张力 = 文章灵魂"是同一个模式。

### Step 4: 生成审查报告

将结果写入文件（不挤主对话上下文）：

```markdown
# Plan Review Report

## 审查对象
- Plan: [文件路径]
- 审查时间: [timestamp]

## 总评
[Go / Go with concerns / Needs rework] — [一句话理由]

## 4 角色审查摘要

### 用户代言人 ✅/⚠️
[一句话结论 + 最重要的 1 个关注项]

### 架构师 ✅/⚠️
[一句话结论 + 最重要的 1 个关注项]

### 风险猎人 ✅/⚠️
[一句话结论 + 最重要的 1 个关注项]

### 务实主义者 ✅/⚠️
[一句话结论 + 最重要的 1 个关注项]

## 高优先级问题（2+ 角色共振）
- [问题1]: [哪些角色提到] → [建议]
- [问题2]: ...

## 交叉盲区
- [盲区1 by 角色X]: ...
- [盲区2 by 角色Y]: ...

## 核心张力
[一句话描述最大分歧，帮用户做权衡判断]

## 建议行动
- [ ] [具体修改建议1]
- [ ] [具体修改建议2]
```

### Step 5: 呈现给用户

在主对话中只显示**精炼摘要**：

```
审查完毕。4 个角色都看过了：

总评：[Go with concerns]
高优先级：[1-2 个共振问题]
核心张力：[一句话]

详细报告在 [文件路径]。
要修改 plan 还是直接开干？
```

## 与流水线的集成

### 接 writing-plans 之后
```
writing-plans 完成
    ↓ 自动触发
multi-role-review
    ↓ 用户确认
subagent-driven-development / executing-plans
```

### 独立使用
用户随时可以说"帮我审查一下这个 plan"，手动触发。

## 配置

### 角色数量可调
默认 4 角色。如果 plan 很简单（< 3 个 task），可以缩减为 2 角色：
- User Advocate + Pragmatist（最常用组合）

### 严格模式
如果任何角色给出 "Needs rework"，自动阻止进入执行阶段，要求先修改 plan。

### 审查报告位置
默认写入 plan 同目录：`{plan-dir}/review-report.md`

## 反模式

- **不要跳过审查因为"plan 很简单"** — 简单 plan 用 2 角色快速过
- **不要忽略核心张力** — 张力就是需要你做判断的地方
- **不要让审查变成走过场** — 如果 4 个角色全说 Go 且没有关注项，要怀疑审查深度
- **不要在主对话贴完整报告** — 写文件，主对话只看摘要
